# Proposal Alignment Note

## 一、本轮实现与开题报告“方法”的对应关系

### 1. 混合检索继续复用

- 本轮没有另起一套 retriever。
- 比较问答仍然复用现有 `HybridRetrievalEngine.retrieve(...)`。
- 变化只在于：从“单 query 单次召回”扩展成“同一比较问题下，对 A / B 分别做定向召回”。

对应到开题报告的方法承诺，本轮仍然站在：

- hybrid retrieval
- dense + sparse 召回
- RRF 融合
- rerank 排序

这些现有能力之上继续做，而不是推翻它们。

### 2. 重排序继续参与

- 比较路径下，A 和 B 两侧都仍然走原有 hybrid + rerank 链路。
- 本轮没有跳过 rerank，也没有改 rerank 模型或阈值。
- 因此比较能力不是“手搓两个 SQL 结果拼起来”，而是继续建立在既有检索排序结果之上。

### 3. 基于证据生成的做法更明确地体现在比较问答里

- 本轮没有引入自由发挥的医学解释。
- 回答中的差异点，优先来自两侧方文的显式文字：
  - `于某方内，加 / 去 ...`
  - 方文中的药味差异
  - 能回指到相关条文时的语境差异
- 这使比较答案更符合“基于证据整理”的方法承诺，而不是把比较问答退化成条文罗列。

### 4. 引用 / 出处机制继续保留

- 比较答案仍然输出 `citations`
- 仍然输出 `primary_evidence / secondary_evidence / review_materials`
- 若某一侧语境只能在 review 层看到，也不会越级进入 primary
- 因此本轮没有破坏“证据溯源”的方法承诺，反而把它延伸到了 A/B 对照场景

### 5. evidence gating 继续是最终裁决器

- `chunks` 仍然不直接进 `primary_evidence`
- `annotations` 仍然不进 `primary_evidence`
- `passages / ambiguous_passages` 仍然只放在 `review_materials`
- 当用户明确追问 `证候 / 条文语境`，而两侧没有足够 safe 正文支撑时，比较答案会降级为 `weak_with_review_notice`

这意味着本轮没有为了“让比较看起来更聪明”而破坏原来最重要的方法约束。

## 二、本轮实现与开题报告“功能”的对应关系

### 1. 系统更接近“研读支持系统”，而不是停留在检索展示

本轮之前：

- 系统面对比较问题，最多只能把两侧条文各自召回出来
- 结果更像“检索展示”

本轮之后：

- 系统能够把两侧证据分开
- 能把显式差异整理成 2~4 条可读的比较结论
- 能在证据不足时降级，而不是硬把比较说成确定答案

因此它比单纯展示检索结果，更接近“研读支持”。

### 2. 更好支持“典型问题案例”

开题报告与范围冻结里强调的是：

- 用户围绕条文内容提问
- 系统提供可检索、可问答、可查看依据的研读支持

“A 和 B 的区别是什么”正是典型的研读型问题。

本轮补上这一类问题后，系统能覆盖：

- 单方原文追问
- 弱证据解释追问
- 无依据拒答
- 最小方剂辨析

这比原来只覆盖“原文是什么 / 这句话什么意思 / 书里有没有”更接近答辩场景。

### 3. 更接近“系统能回答条文解释 / 方剂辨析类问题”

- 条文解释类问题：原有 weak/strong 路径已可支持
- 方剂辨析类问题：本轮新增 comparison answer 后开始具备最小可用性

尤其是目标问题：

- `桂枝加附子汤方和桂枝加厚朴杏子汤方有什么区别`

现在系统不再只是罗列两条方文，而能输出：

- 一句总述
- 显式加味差异
- 相关条文语境线索
- 出处线索
- 并保留 citations / evidence 分层

## 三、本轮补齐了什么

- 补齐了最小 pairwise comparison 能力
- 补齐了“方名 vs 方名”的最小辨析答案
- 补齐了 comparison-specific 的强弱裁决
- 补齐了 comparison 样例与 smoke check
- 补齐了与开题报告方法 / 功能承诺的单独对齐说明

## 四、还没补齐什么

仍未完全落实的内容包括：

- 多书比较
- 三方以上比较
- 更完整的 formula alias / title normalization
- 更稳定的方文与相关主条映射
- 更结构化的前端对照展示
- 历史记录、用户系统、复杂交互

## 五、当前不补的理由

- 这些内容都超出本轮“最小实现”原则
- 当前最关键的缺口，是把系统从“会检索、会展示”推进到“能给出最小辨析答案”
- 若继续扩展到多书、知识图谱、复杂前端或更大规模别名工程，会直接稀释本轮目标，也违背当前冻结范围

## 六、结论

### 1. 这轮完成后，系统是否更接近“开题报告中承诺的研读支持功能”？

是。

原因不是“功能变多了”，而是：

- 它开始能对典型研读问题给出整理后的差异答案
- 仍然保留证据分层与出处
- 证据不足时仍然会降级

因此它更接近“研读支持系统”，而不是只停留在“检索展示页面”。

### 2. 如果老师现场问“桂枝加附子汤方和桂枝加厚朴杏子汤方有什么区别”，系统现在能否给出更像研读支持答案的输出？

可以。

它现在至少能做到：

- 识别两个方名
- 分侧组织证据
- 明确给出显式加味差异
- 给出相关条文语境线索
- 保留 citations 与核对提示

这已经比单纯把两条方文堆出来，更像“研读支持答案”。
