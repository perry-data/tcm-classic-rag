# 前端证据标记展示体验优化 (v1)

## 背景与目标
在系统之前的版本中，为了保证回答的可核验性，后端 validator 强制在回答正文 (`answer_text`) 每一行的末尾附带原始的证据引用标记（如 `[E1][E2]`）。虽然这严格保证了引用的准确对应，但也带来了视觉噪音，使用户在阅读正文时容易被打断。

本轮优化仅在**前端呈现层**进行了改动，旨在提升阅读体验，同时兼顾原始审计需求，且严格遵循了不改动后端逻辑、负载契约与 validator 规则的约束。

## 核心变更点

1. **内联轻量级徽标展示 (Chips)**
   - 拦截并解析了 `answer_text` 中出现的 `[E#]`。
   - 默认将这些纯文本标记替换为轻量级的小号、浅色徽标（Chip），以脚注序号的风格呈现。正文不再被笨重的方括号撑满，降低了阅读时的视觉干扰。

2. **交互式证据定位与高亮**
   - 徽标支持鼠标悬停（Hover）：利用已有的 `payload` 信息，在 `title` 属性中渲染对应 evidence 的标题与简短摘要 (snippet)。
   - 徽标支持点击跳转：点击行内的任意证据徽标，页面将平滑滚动到主依据列表中对应的 Evidence 卡片，并短暂高亮（2秒）该卡片，使用户能极快地完成溯源核对。

3. **提供原始证据标记的显隐开关**
   - 为了方便答辩演示以及确保系统在审计时的透明度，在聊天界面顶部的工具栏右侧新增了一个非常克制的显隐开关 `[E]`。
   - 该开关默认关闭，点击后可切换到“显示原始证据标记”的状态，以最真实的数据面貌呈现给专业核验人员。

4. **自动证据序号映射映射 (Sequential ID Mapping)**
   - 新增映射机制，前端能够按严格的顺序给 `primary_evidence`、`secondary_evidence` 以及 `review_materials` 依次分发 `e_id`，确保与 `answer_text` 里按顺序生成的 `[E#]` 正确绑定。强模式下自然只关联主依据，非强模式则关联其相应的引用内容。
