# formula_effect_query_hotfix patch note

## 变更范围

本轮只修“方剂作用类问题被错误降格为 source_lookup / 方文直出型回答”的问题，不改 retrieval 主体，不改 payload contract，不改前端，不改 evaluator 主链。

## 代码变更

- 在 `backend/answers/assembler.py` 中新增单方剂“作用类”问法识别，覆盖：
  - `X方有什么作用`
  - `X主什么 / 主治什么`
  - `X适用于什么情况 / 用在什么情况`
  - `X是治什么的 / 有什么用 / 是干什么的`
- 对命中的方剂作用类 query，新增专用 answer assembly：
  - 先输出书中直接使用语境的概括
  - 再给直接条文依据
  - 方文只作为补充，不再喧宾夺主
- 对证据边界做收束：
  - 若只拿到 review/passages 层的直接语境，不再硬装 `strong`
  - 改为 `weak_with_review_notice`，并明确“只能保守理解”
- 收紧方剂语境匹配：
  - 排除“其他方的方文标题行”伪命中
  - 对作用类优先选同章、含 `主之` 的直接语境材料，避免短方名被别方校勘或长方名拖偏
- 额外补一个很窄的组成类护栏：
  - `X方由什么组成` 继续按方文组成作答
  - 只用于守住 query 类型边界，不扩本轮修复范围

## 最小验证

- `竹叶石膏汤方有什么作用？`
  - `answer_mode = strong`
  - 回答先给“伤寒解后，虚羸少气，气逆欲吐”这一直接使用语境，再附条文和补充方文
- `黄连汤方有什么作用？`
  - `answer_mode = weak_with_review_notice`
  - 只按 review 层可核对的“伤寒胸中有热，胃中有邪气，腹中痛，欲呕吐”保守解释，不强装 `strong`
- `竹叶石膏汤方的条文是什么？`
  - 继续按条文 / 方文直读输出，未被误改为作用解释
- `竹叶石膏汤方由什么组成？`
  - 继续按方文组成输出，未被误改为作用解释

## 根因结论

本轮根因是“两者都有，但以 answer assembly 模板错位为主”：

- query intent 识别层：原链路只能识别“这是个方名问题”，不能继续细分成“条文 / 组成 / 作用”
- answer assembly 层：一旦落到标准 strong 路径，就默认使用“直接相关主条如下”的 source_lookup 模板，导致“有什么作用”被答成方文直出

## 非本轮范围

- 不处理 retrieval 主链的通用重排策略
- 不处理 evaluator 题型定义
- 不扩展到现代临床功效总结
- 不把所有方名相关问题合并成一种回答
