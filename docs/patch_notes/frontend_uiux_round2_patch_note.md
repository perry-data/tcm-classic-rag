# Frontend UI/UX Round 2 Patch Note

## 本轮目标

本轮在“阶段性渐进反馈”已经接通的基础上，继续做前端第二轮最小优化。

范围只限于当前单页前端：

- 强化 `strong / weak_with_review_notice / refuse` 三种模式的视觉边界
- 补齐错误态 / 超时态 / 重试体验
- 优化主依据、补充依据、核对材料、引用的阅读层级
- 让完成态收尾更稳，不留 loading / streaming 残影

## 本轮未改动的内容

- 未改后端 retrieval / rerank / prompt 主逻辑
- 未改 answer payload 顶层字段语义
- 未改 evaluator / goldset / 评估逻辑
- 未引入新的前端框架
- 未做整站视觉重设计

## 主要交互变化

### 1. 模式边界强化

- `strong`
  - 顶部显示“可直接参考的回答”
  - 明确提示“先看回答，再核对主依据”
- `weak_with_review_notice`
  - 顶部显示“需核对的回答”
  - 明确说明“不能视为确定结论”
  - `review_notice` 仍单独高亮，不埋进正文
- `refuse`
  - 顶部显示“当前不支持这样回答”
  - 明确指出这是业务拒答，不是系统报错
  - 拒答原因和改问建议保持清楚分区

### 2. 错误态 / 超时态 / 重试

- 流式请求失败后会先尝试回退到同步接口
- 长时间等待会显示“仍在继续处理”的软提示
- 超时后会主动结束当前请求，防止页面一直停在“正在生成”
- 流式和同步都失败时，会进入独立错误区
- 错误区包含：
  - 错误标题
  - 错误说明
  - “这不是拒答”的说明
  - “重新提交上一个问题”按钮

### 3. 证据区层级

- 主依据：视觉最强，卡片强调更明显
- 补充依据：保留阅读价值，但弱于主依据
- 核对材料：默认收起，避免抢走主回答焦点
- 引用：用更直观的“回答引用”命名，并按角色做弱区分
- evidence card / citation item 统一显示：
  - 证据角色
  - 证据等级
  - 来源类型
  - 记录简写
  - chapter 信息
  - 完整 `record_id`

### 4. 完成态收尾

- 完成后 progress 区切到低强调状态
- 清除长等待提示
- 清除 streaming 光标
- 避免完成后继续保持“处理中”视觉焦点

## 本轮新增的交互状态

本轮新增但不改变 payload 语义的前端状态包括：

- `loading`
- `error`
- 长等待提示
- 流式失败后的同步回退中
- review 折叠 / 展开

这些都属于展示态或 transport 态，不属于业务 answer mode。

## 本轮验证

### 强结果

- query：`黄连汤方的条文是什么？`
- 结果：`strong`
- 验证点：
  - `primary_evidence` 仍存在
  - `review_notice` 仍存在
  - 页面将按“可直接参考的回答”展示

### 弱结果

- query：`烧针益阳而损阴是什么意思？`
- 结果：`weak_with_review_notice`
- 验证点：
  - `primary_evidence = []`
  - `review_notice` 仍存在
  - 页面将按“需核对的回答”展示

### 拒答结果

- query：`书中有没有提到量子纠缠？`
- 结果：`refuse`
- 验证点：
  - `refuse_reason` 仍存在
  - `suggested_followup_questions` 仍存在
  - 页面将按“当前不支持这样回答”展示，而不是错误态

### 错误场景

- 方式：前端 transport failure 模拟
- 模拟结果：
  - 流式请求失败
  - 同步回退再次失败
  - 页面进入独立错误态
  - 错误标题为“流式与标准请求都未完成”
  - 重试按钮可用
  - 进度区不再停留在“正在生成”

## 改前 / 改后差异

### 改前

- 模式区分主要依赖 badge 和 callout
- 错误态不够完整，失败后缺少明确重试出口
- 核对材料和主依据的视觉层级仍偏接近
- 完成后 progress 区仍容易抢视线

### 改后

- 模式边界由“颜色 + badge”升级为“标题 + 解释 + 阅读建议”
- 错误态与拒答态彻底分开
- 长等待、超时、回退、失败都能收尾
- 主依据更突出，核对材料默认收起
- 完成后页面更稳，更接近正式作品展示
