# definition_outline_query_hotfix patch note

## 变更范围

本轮只修“定义 / 提纲类问题”的主依据裁决错误，不改 retrieval 主体、不改 payload contract、不改前端、不改 evaluator 主链，也不把这轮扩成通用检索重写。

## 代码变更

- 在 `backend/answers/assembler.py` 中新增很窄的 `DefinitionOutlinePlan` 分流，显式识别：
  - `X是什么？`
  - `X是什么病？`
  - `X之为病是什么？`
  - `X的提纲是什么？`
  - `X的定义是什么？`
- 本轮先把 `X` 收窄到六经病主轴：
  - `太阳病 / 阳明病 / 少阳病 / 太阴病 / 少阴病 / 厥阴病`
- 对命中的定义 / 提纲类 query，新增专用 evidence prioritization：
  - 优先扫描 `main_passages` 中承担定义 / 总纲作用的句式
  - 显式提升 `X之为病……`
  - 兼容 `X之病……`、`凡X……`、`X病……名曰 / 名为……`、`问曰：X病……答曰……` 这类弱一档定义句
  - 对只因 `X病` 三字连续连写而高分的普通治法条文，保留为补充依据，不再天然占据主依据首位
- 在定义句清洗里补了一个很窄的校勘兼容：
  - 将 `少阳之赵本医统本皆有「为」字病……` 还原为可识别的 `少阳之为病……`
  - 只用于定义 / 提纲类候选识别与 answer_text 组织
- 定义 / 提纲类回答模板改为：
  - 先说“书中对该概念的提纲性表述”
  - 再给主依据条文
  - 其余相关条文降到补充依据

## 最小验证

- `太阳病是什么？`
  - 主依据首位改为：`太阳之为病，脉浮，头项强痛而恶寒。`
- `阳明病是什么？`
  - 主依据首位改为：`阳明之为病，胃家实是也。`
- `少阳病是什么？`
  - 主依据首位改为：`少阳之为病，口苦、咽乾、目眩也。`
- `太阴病是什么？`
  - 主依据首位改为：`太阴之为病，腹满而吐，食不下，自利益甚，时腹自痛。若下之，必胸下结硬。`
- `太阳病应该怎么办？`
  - 仍走 general question / 分支整理路径，未被误改为定义句回答
- `太阳病的条文是什么？`
  - 仍走条文类直出路径，未被误改为定义 / 提纲类回答

## 根因结论

本轮根因是“两层都有，但以 query intent 缺失触发了 evidence prioritization 错位”为主：

- query intent 识别层：
  - 原链路没有把 `X是什么？` 这类定义 / 提纲类问题单独识别出来
  - 于是它们落回普通 strong/source lookup 路径
- candidate rerank / evidence prioritization 层：
  - 普通检索更偏好 `X病` 连续词面命中
  - `X之为病` 这类真正总纲句因为词面不连续、且常是短句，容易被压到后面

因此本轮修复的核心不是“补答案文案”，而是：

- 先识别 definition / outline intent
- 再在 assembler 层把定义句模板抬成主依据

## 非本轮范围

- 不处理所有非六经病概念的“是什么”问法
- 不处理现代病理解释或跨书定义比较
- 不处理 general overview / source lookup 的通用排序重构
- 不改 evaluator 题型定义与 payload 结构
