# Cleanup Plan v1

## 1. 本轮定位

本轮目标不是“立即大整理”，而是先把“下一步怎么整理、为什么现在不乱动”写清楚。

因此，本方案坚持三条原则：

- 不破坏现有可运行入口
- 不大规模移动文件
- 先收口规则，再做物理归档

## 2. 当前文件夹混乱的根因

当前混乱不是单点问题，而是几类文件长期叠加的结果：

1. 根目录承担了太多角色：
   - 运行入口
   - patch note
   - spec
   - 盘点清单
   - 临时对齐说明
2. `artifacts/` 中既有正式运行资产，也有历史 smoke 结果，但没有统一口径说明。
3. `backend/`、空文档、空占位目录仍在仓库里，可见但不参与运行。
4. “哪些文件是正式基线”此前没有明文清单，导致 patch note 容易反客为主。

## 3. 目录保留建议

以下目录建议保留，并继续作为主分层使用：

| 目录 | 建议 | 原因 |
| --- | --- | --- |
| `docs/` | 保留并继续强化 | 文档统一收口地已经存在 |
| `artifacts/` | 保留 | 构建产物、smoke、examples 已集中在这里 |
| `scripts/` | 保留 | 数据前置处理脚本已在这里 |
| `frontend/` | 保留 | 当前前端入口清晰，且和 API 同源 |
| `data/` | 保留 | 原始 / 处理后数据分层已经明确 |
| `dist/` | 保留 | safe 数据包的自然位置 |
| `reports/` | 保留 | 结构化验收报告的自然位置 |

## 4. 是否建议创建更清晰分层

### `docs/`

建议：`是`

说明：

- 已存在，应继续作为所有正式文档的统一收口位置。
- 后续不再新增根目录散落 spec / patch note 作为主文档。

### `artifacts/`

建议：`是`

说明：

- 已存在，应继续作为构建产物、smoke 和 examples 的统一目录。
- 现阶段不建议再把运行产物写回 `data/indexes/` 或 `outputs/`。

### `scripts/`

建议：`是`

说明：

- 数据验收和 safe 数据构建已在这里。
- 但当前 `build_mvp_database.py`、`build_dense_index.py`、`run_*` 还在根目录；本轮不建议立即全部移动过去。

### `app/`

建议：`暂不创建为正式运行层`

说明：

- 虽然从工程美观角度看，把运行逻辑放进 `app/` 更整齐，但当前最小 API 入口就是 `app_minimal_api.py`。
- 现在把运行逻辑整体搬进包结构，会引入路径、导入、启动方式的连锁风险。
- 对当前毕设封板价值不高，应推迟到封板后再评估。

### `frontend/`

建议：`继续保留`

说明：

- 当前三文件结构简单稳定，适合答辩与维护。

### `data/`

建议：`继续保留`

说明：

- `raw/`、`processed/` 已具备最基本的数据分层。

## 5. 哪些文档建议统一收口到 `docs/`

后续新增的以下文档类型，都应优先写进 `docs/`：

- 正式 PRD
- tech spec
- repo map / inventory
- change workflow
- cleanup plan
- 新功能 change note
- 后续答辩口径文档

当前已经在根目录的 patch / note / spec，不建议继续在那里扩写。

## 6. 哪些 artifacts 建议集中放置

当前 `artifacts/` 已基本承担此职责，建议继续保持：

- `*_smoke_checks.md`
- `*_examples.json`
- `database_*`
- `dense_*`
- `zjshl_mvp.db`

处理原则：

- 运行资产继续放 `artifacts/`
- 验证资产继续放 `artifacts/`
- 不在根目录新增新的 smoke / example 文件

## 7. patch note / smoke check 应如何归档

### patch note

本轮建议：

- 先不移动现有根目录 patch note
- 但从现在开始，根目录 patch note 不再视为主文档

下一阶段建议：

- 封板后把历史 patch note 统一归档到 `docs/history/` 或等价目录
- 归档前先完成一轮链接核对，避免引用断裂

### smoke check

本轮建议：

- 继续放在 `artifacts/`
- 不做文件名大改

下一阶段建议：

- 若数量继续增加，可再细分为：
  - `artifacts/smoke/`
  - `artifacts/examples/`
  - `artifacts/build/`

但这一步应推迟到封板后做。

## 8. 哪些移动现在就能做

本轮只建议做“零风险收口”，不建议做结构性搬迁。

现在就能做的只有：

1. 把正式基线文档写入 `docs/`
2. 明确以后新增正式文档优先进入 `docs/`
3. 明确 `artifacts/` 是运行 / 验证产物目录
4. 明确根目录 patch note 以后不再承担主规格角色

也就是说，本轮“能做的移动”主要是规则层面的，而不是大规模物理移动。

## 9. 哪些移动应该推迟到封板后做

以下动作建议推迟：

1. 把根目录 `run_*` / `build_*` 全部搬进 `scripts/` 或 `app/`
2. 把根目录 patch note / spec / inventory 统一迁到 `docs/history/`
3. 清理 `backend/` 空壳 scaffold
4. 清理零字节文档和占位目录
5. 合并重复虚拟环境，仅保留 `.venv/`
6. 把 `artifacts/` 细分成多级子目录

推迟原因：

- 这些动作会改路径
- 会影响已有命令、文档、答辩口径和人工记忆
- 对当前“先封板、再答辩整理”的收益不高

## 10. 当前最小整编方案

### 现在做

- 建立本轮基线文档包
- 冻结当前正式入口和正式样例
- 明确 canonical / supporting / legacy
- 停止继续把新规格散落到根目录

### 暂不做

- 不搬运行入口
- 不重命名主路径
- 不删大量文件
- 不重构为新包结构
- 不修改核心问答逻辑

## 11. 最终建议

当前项目最适合的整理方式不是“立刻重构目录”，而是：

1. 先把正式基线文档化
2. 让后续开发先服从基线
3. 等封板后，再做小规模归档和去噪

这比现在一边加功能、一边移动文件更安全，也更符合毕设项目的答辩与论文节奏。
