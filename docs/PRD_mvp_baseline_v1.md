# PRD MVP Baseline v1

## 1. 文档定位

- 文档目标：冻结当前毕业设计项目的正式 MVP 产品边界，给后续开发、论文映射、答辩说明和变更控制提供统一依据。
- 适用对象：后续 coding agent、项目维护者、论文撰写者、答辩审查者。
- 基线口径：以当前仓库真实实现为准，不以理想蓝图为准。
- 当前正式闭环：`query -> hybrid retrieval -> evidence gating -> answer assembler -> POST /api/v1/answers -> 同源前端单页渲染`
- 重要说明：仓库中已经出现比较问答、总括问答等扩展实现，但本 PRD v1 不把它们写入正式 MVP 承诺；它们只能视为现存扩展代码，不作为本轮范围和成功标准。

## 2. 项目目标

本项目要完成的是一个面向《伤寒论》单书研读场景的最小可用系统。系统接收自然语言问题，在固定数据底座内做检索，按证据分层组织回答，并把可见依据与回答一起返回。

本项目的目标不是提供“更聪明的中医大模型”，而是验证以下命题在单书场景中的可行性：

- 检索增强可以降低脱离原文的回答风险。
- 证据分层和证据溯源可以让回答更可核对。
- 当证据不足时，系统应弱回答或拒答，而不是强行给出确定性结论。

## 3. 用户是谁

当前目标用户是围绕《伤寒论》做文本研读的人，而不是临床问诊用户：

- 中医药相关专业学生
- 课程教学或读书讨论中的教师、研究者
- 需要快速定位条文与依据的研读型使用者

默认用户接受以下前提：

- 系统是研读支持工具，不是诊疗工具。
- 系统输出的可信度以“能否回到原文依据”来判断。
- 当系统提示“需核对”或“拒答”时，这不是故障，而是设计要求。

## 4. 系统要解决什么问题

当前系统聚焦解决以下实际问题：

- 用户提出自然语言问题后，人工翻书或逐条查找效率低。
- 仅靠生成式回答容易脱离《伤寒论》原文，缺少可核对依据。
- 研读问题常常不是“有没有答案”，而是“有没有足够依据支撑答案”。
- 数据底座存在分层与风险差异，系统需要把强证据、辅助材料、核对材料区分开。

因此，本系统提供的是“可检索、可回答、可引用、可拒答”的研读支持能力，而不是权威结论输出能力。

## 5. 当前 MVP 功能边界

### 5.1 当前正式承诺的能力

本版正式承诺以下能力：

1. 单书范围固定：只围绕《伤寒论》工作。
2. 单轮问答：输入一个问题，返回一个 answer payload。
3. 检索后作答：回答建立在检索结果之上，不走脱离证据的自由生成。
4. 证据分层：区分 `primary_evidence`、`secondary_evidence`、`review_materials`。
5. 三种输出模式：
   - `strong`
   - `weak_with_review_notice`
   - `refuse`
6. 最小 API：前端只依赖 `POST /api/v1/answers`。
7. 最小前端：同源单页页面展示问题、回答、依据、核对提示、拒答原因和改问建议。

### 5.2 当前明确不做什么

本版明确不做以下内容：

- 多书扩展
- 诊疗建议、辨证论治、处方推荐、剂量建议
- 用户系统、历史记录、收藏分享
- 大规模前端增强
- payload 合同改版
- API 合同推翻
- 检索器大改或数据库方案重做
- 知识图谱、多 agent、复杂工作流

### 5.3 本版不纳入正式范围但仓库中已存在的内容

以下内容在仓库里有实现痕迹，但不属于本 PRD v1 的正式承诺：

- 比较类问答
- 总括类 / 泛问类问答
- 这些功能相关的 patch note、check 脚本和 smoke 产物

处理原则是：

- 可以保留
- 可以参考
- 不能作为本轮正式验收依据
- 后续若要继续做，必须先走 `change_workflow_v1`

## 6. 关键用户路径

### 路径 A：直接条文 / 方名查询

1. 用户输入一个具体问题，例如“黄连汤方的条文是什么？”
2. 系统执行 hybrid retrieval，并按证据门控回指到安全主条。
3. 系统返回 `strong` 模式 payload。
4. 前端展示 `answer_text`、主依据、补充依据和引用。

### 路径 B：有线索但强证据不足

1. 用户输入一个研读问题，例如“烧针益阳而损阴是什么意思？”
2. 系统检索到辅助材料或风险材料，但没有足够主证据。
3. 系统返回 `weak_with_review_notice`。
4. 前端强调“需核对”，不把结果渲染成确定答案。

### 路径 C：超出书内依据或无有效命中

1. 用户输入书内无依据的问题，例如“书中有没有提到量子纠缠？”
2. 系统未命中足够证据。
3. 系统返回 `refuse`。
4. 前端展示拒答原因和改问建议，而不是报错页。

## 7. 典型问题类型

当前正式 MVP 主要面向以下问题：

| 类型 | 说明 | 当前预期 |
| --- | --- | --- |
| 方名 / 条文直问 | 直接问某方、某条文、某句原文 | 优先走 `strong` |
| 局部含义研读 | 问某句什么意思、如何理解 | 可能 `strong`，也可能 `weak_with_review_notice` |
| 书内存在性询问 | 问书中是否提到某概念 | 有依据则回答，无依据则 `refuse` |
| 缩小范围的核对问题 | 围绕某一条或某一名称继续追问 | 允许，前提是仍在单书范围内 |

以下问题类型不作为本版正式承诺：

- 比较类问答
- 总括类问答
- 书外常识问答
- 临床处置与治疗建议问答

## 8. 系统输出形式

当前系统的正式输出不是“纯文本答案”，而是稳定的 answer payload。对用户可见的主要内容包括：

- `answer_text`
- `answer_mode`
- `primary_evidence`
- `secondary_evidence`
- `review_materials`
- `review_notice`
- `refuse_reason`
- `suggested_followup_questions`
- `citations`

也就是说，本系统的产品形式本质上是“带证据槽位的回答对象”，前端只是其消费层。

## 9. 成功标准

本 PRD v1 的成功标准不是“功能越多越好”，而是以下事项同时成立：

1. 用户能通过同源 Web 页面完成单轮提问。
2. 页面只依赖 `POST /api/v1/answers`，不依赖检索内部 trace。
3. 系统能稳定输出三种模式：`strong`、`weak_with_review_notice`、`refuse`。
4. `primary_evidence` 与辅助 / 风险材料保持分层，不混用。
5. 证据不足时系统会收缩回答，而不是强答。
6. 原三条冻结样例保持成立：
   - `黄连汤方的条文是什么？ -> strong`
   - `烧针益阳而损阴是什么意思？ -> weak_with_review_notice`
   - `书中有没有提到量子纠缠？ -> refuse`
7. 系统对外定位始终保持“单书研读支持、非诊疗”。

## 10. 非目标范围

以下内容即使技术上已经出现代码雏形，也不属于本版正式目标：

- 把仓库中已有扩展代码直接升级成正式需求
- 通过新增功能数量证明项目成熟度
- 通过大规模目录重构追求“看起来更像产品工程”
- 以比较问答、总括问答、答辩增强页替代当前 MVP 封板工作
- 为了“更聪明”而放松当前 evidence gating

## 11. 本版 PRD 的使用规则

- 后续新增功能前，先看本 PRD 是否已经覆盖；未覆盖则不能直接开做。
- 若新增功能改变了“用户是谁、系统做什么、当前承诺什么、成功标准是什么”，必须先更新本 PRD。
- 论文、答辩、开发说明中，对外描述应以本 PRD v1 为正式口径。
