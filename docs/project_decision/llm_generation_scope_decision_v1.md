# 真实 LLM API / 分步 Prompt / React 前端专项决策 v1

- 决策日期：2026-04-09
- 输入依据：`docs/project_audit/proposal_vs_actual_gap_audit_v1.md`、`docs/project_audit/proposal_vs_actual_gap_matrix_v1.json`、`docs/final/PRD_v1.md`、`docs/final/technical_design_v1.md`、`docs/final/system_spec_v1.md`、`docs/evaluation/evaluation_spec_v1.md`、当前 `AnswerAssembler` / `Minimal API` / `frontend MVP` / `evaluator v1` / `goldset 150`
- 决策范围：只判断要不要补做，不做实现

## 1. 结论先行

这三项不能打包成一件事处理。它们对题目成立度、论文收益、实现成本和稳定性风险的影响完全不同。

| 项目 | 最终结论 | 一句话判断 |
| --- | --- | --- |
| 真实 LLM API 接入 | 可做可不做，但若不做需在论文中改口径 | 这是三项里唯一直接影响“生成”成立程度和答辩时“你这到底是不是 RAG”的项。 |
| 分步 Prompt / Prompt 组装链路 | 不必补做 | 它不是独立必要项，只有在真实 LLM 已接入且稳定后才有价值。 |
| React + Tailwind 前端重写 | 不必补做 | 它几乎不影响 RAG 成立，也不提升第 4 章说服力，工程收益最低。 |

补做优先级只看这三项时：

1. 真实 LLM API
2. 分步 Prompt / Prompt 组装链路
3. React + Tailwind 前端重写

若只补一个，最应该补的是：**真实 LLM API 的最小接入版**。
但推荐形态不是“推翻现有 AnswerAssembler 重做一遍”，而是“保留 retrieval / gating / citations / payload contract，只让 LLM 负责受约束的 `answer_text` 生成或润色”。

## 2. 决策标准

本决策按五个维度判断：

1. 是否影响题目“基于检索增强生成”的成立程度。
2. 是否影响论文第 2/3/4 章的说服力。
3. 是否影响答辩时老师对“这是不是 RAG”的直觉判断。
4. 实现成本有多高。
5. 对当前 MVP 稳定性冲击有多大。

当前项目的现实约束也必须一起看：

- 当前正式主链路已冻结为 `query -> hybrid retrieval -> evidence gating -> answer assembler -> POST /api/v1/answers -> frontend`
- `AnswerAssembler` 已有约 2100+ 行规则与分流逻辑，不是一个空壳
- `Minimal API`、前端字段消费、evaluator v1 都围绕当前 answer payload contract 和三模式语义建立
- `evaluation_spec_v1` 明确写了：当前 v1 评估**不覆盖真实 LLM / Prompt 栈效果**

所以，这三项不是“多写一点代码”这么简单，而是要不要打破当前稳定评估边界的问题。

## 3. 分项决策

### 3.1 真实 LLM API 接入

#### 当前现状

当前系统的“生成”是 `AnswerAssembler` 的证据驱动规则化编排，不是外接 LLM。`PRD_v1.md`、`technical_design_v1.md`、`system_spec_v1.md` 都已明确这一点。

#### 对题目成立度的影响

影响高。

原因很直接：题目里写的是“检索增强生成”。当前系统当然已经有检索增强，但“生成”更接近模板化回答编排，而不是通常意义上的 LLM generation。严格一点的老师会接受“这是 evidence-driven answer assembler”，但也很可能继续追问：“那生成在哪？”

所以这项是三者里**最直接影响题目字面成立程度**的一项。

#### 对论文第 2/3/4 章的影响

- 第 2 章：影响中高。是否有真实 LLM，会直接影响“答案生成模块设计”怎么写。
- 第 3 章：影响高。是否存在模型调用、超时处理、输出约束、回退逻辑，会改变实现章节结构。
- 第 4 章：影响中高。若接入真实 LLM，当前 evaluator v1 就不再覆盖全部关键行为，需要额外说明 nondeterminism、提示约束与失败样本。

#### 对答辩“是不是 RAG”的影响

影响高。

如果不接真实 LLM，答辩时最稳的说法只能是：

“当前系统完成了 retrieval-augmented evidence organization / answer assembly，生成部分收口为规则化编排。”

这并不等于不能答辩，但确实少了一层直观说服力。
如果接入真实 LLM，即使只是最小接入版，也能更自然回答“为什么叫 RAG”。

#### 实现成本

成本高。

真正的成本不在“调个 API”本身，而在以下连锁工作：

1. 选模型与调用方式。
2. 增加配置、密钥、超时、失败回退。
3. 设计受约束的输入上下文，避免模型绕开现有 evidence gating。
4. 保证 `answer_mode`、`citations`、三槽位语义不被模型输出冲乱。
5. 重新做 smoke / regression，必要时补一轮 LLM 专项评估说明。

如果做成“LLM 自由生成 + 自己决定引用”，成本和风险都非常高。
如果做成“LLM 只消费固定证据槽位，并只负责 `answer_text` 的受约束生成”，成本会降一些，但仍明显高于另外两项。

#### 对 MVP 稳定性的影响

风险高。

当前 MVP 的稳定性来自三个东西：

1. 三模式语义稳定。
2. evidence slots 稳定。
3. evaluator v1 可以直接复跑。

真实 LLM 一旦接入，最大的风险不是页面挂掉，而是：

- strong / weak / refuse 边界被冲淡
- 模型把 secondary / review 说成主结论
- 引用文本与 answer_text 出现语义漂移
- 同题多跑结果波动，影响第 4 章复现性

#### 论文收益

收益高，而且是三项里最高。

它补的是“生成”这一层，不是 UI 面子工程。
如果只从论文与答辩收益比来看，三项里最值得补的就是它。

#### 最终结论

**可做可不做，但若不做需在论文中改口径。**

当前冻结范围下，它不是必须补做；但它是唯一一个如果不补，就必须明确改写论文口径的项。

#### 若决定补，推荐的最小形态

不要重写整个系统。推荐最小形态是：

1. retrieval / rerank / evidence gating / citations / answer_mode 全保留。
2. `AnswerAssembler` 仍然负责模式裁决、证据分层和 citation。
3. LLM 只基于固定的 `primary / secondary / review` 输入生成或润色 `answer_text`。
4. 一旦 LLM 超时、失败或越界，自动回退当前规则化答案。

这样补，论文收益最大，爆炸半径最小。

### 3.2 分步 Prompt / Prompt 组装链路

#### 当前现状

当前没有真实 prompt 栈。现有“控制意图”主要体现在规则模板、三模式裁决和 evidence gating 上。

#### 对题目成立度的影响

影响低到中。

Prompt 工程会影响“生成质量怎么更稳”，但不会像真实 LLM API 那样直接决定“有没有生成模块”。

换句话说：

- 没有真实 LLM，分步 Prompt 基本无从谈起。
- 有了真实 LLM，也不是必须做成多步 prompt 才算 RAG。

#### 对论文第 2/3/4 章的影响

- 第 2 章：影响中。会影响“提示模板设计”和“创新点”写法。
- 第 3 章：影响中。会多一个 prompt orchestration 章节，但不是系统成立的必要骨架。
- 第 4 章：影响中。多步 prompt 会引入新的变量，反而让评估口径更复杂。

#### 对答辩“是不是 RAG”的影响

影响中低。

老师先问的通常不是“你是不是多步 Prompt”，而是：

1. 有没有 retrieval
2. 有没有 generation
3. 有没有 evidence grounding

所以这项对“是不是 RAG”的帮助明显弱于真实 LLM API。

#### 实现成本

成本中高。

多步 Prompt 不是“多写几个字符串”，而是要决定：

1. 每一步输入什么。
2. 每一步输出什么结构。
3. 上一步错误如何传递。
4. 多步之间如何防止引用漂移。
5. 多步链路如何测延迟和失败回退。

而当前 `evaluation_spec_v1` 又明确不覆盖真实 LLM / Prompt 栈效果，意味着一旦补做，还要额外补评估说明。

#### 对 MVP 稳定性的影响

风险中高。

分步 Prompt 比单步 Prompt 更容易出现：

- 上下游步骤不一致
- 证据释义与最终回答脱节
- 延迟叠加
- debug 难度上升

对当前已经冻结的 MVP 来说，这是典型的“复杂度上升快于论文收益”的项。

#### 论文收益

收益中等。

它主要提升的是“方法描述的漂亮程度”和“创新点的技术味道”，不是系统可信度的主来源。
如果真实 LLM 都不接，直接补分步 Prompt 基本没有独立收益。
如果真实 LLM 已接入，先用单个强约束 prompt 往往就足够支撑毕设级说明。

#### 最终结论

**不必补做。**

它不是独立必要项，也不应该在真实 LLM 尚未接入时单独推进。
若后续真的决定补 LLM，优先顺序也应该是：

1. 先接最小 LLM API
2. 先用单个受约束 prompt 跑通
3. 只有在结果明显不稳时，才考虑多步 Prompt

### 3.3 React + Tailwind 前端重写

#### 当前现状

当前前端是原生 HTML/CSS/JS 同源 SPA，字段消费和交互逻辑都已围绕冻结 answer payload contract 跑通；`frontend/app.js` 只有约 280 行，结构并不失控。

#### 对题目成立度的影响

影响极低。

这项几乎不影响“基于检索增强生成”的成立程度。
老师不会因为你用了原生 JS 而否认它是不是 RAG。

#### 对论文第 2/3/4 章的影响

- 第 2 章：影响低。只是技术选型表述变化。
- 第 3 章：影响中低。会让“前端界面实现”看起来更现代，但不改变系统主链。
- 第 4 章：几乎无正向收益。测试说服力主要来自 retrieval、evidence、answer mode、citations，不来自 React。

#### 对答辩“是不是 RAG”的影响

影响极低。

React / Tailwind 影响的是 UI 工程风格，不影响检索增强问答的学术判断。

#### 实现成本

成本中等。

单看页面规模，重写不算天文成本；但真实成本在于：

1. 建构建工具链
2. 加包管理
3. 处理同源部署与静态资源路径
4. 重写前端 smoke
5. 重写前端实现章节和运行说明

也就是说，它不是“功能更强”，而是“换一套工程壳”。

#### 对 MVP 稳定性的影响

风险中等。

当前前端最大价值是已经稳定、已经和现有 payload 对齐。
重写为 React + Tailwind 的风险不在功能逻辑本身，而在：

- 新增构建链路
- 静态资源部署变化
- smoke 基线失效
- 临近收尾时引入纯工程层回归

#### 论文收益

收益低。

它对 UI 展示会有帮助，但对论文最关键的第 4 章结果、对“是不是 RAG”的判断、对研究方法的可信度，几乎没有核心增益。

#### 最终结论

**不必补做。**

除非答辩或学院评分明确把前端框架当成硬指标，否则这项不值得在当前阶段重写。

## 4. 推荐顺序

### 4.1 只看这三项时的推荐顺序

1. 真实 LLM API 接入
2. 分步 Prompt / Prompt 组装链路
3. React + Tailwind 前端重写

### 4.2 更实际的执行建议

在当前项目整体收尾顺序里，这三项都不应排在“图件、部署运行支撑、响应速度测试、用户满意度最小证据”之前。

也就是说：

- 这三项里，只有真实 LLM API 值得考虑补。
- 分步 Prompt 只有在 LLM 已接入后才值得讨论。
- React + Tailwind 重写应直接放到答辩后或不做。

## 5. 若只补一个，最应该补哪个

答案：**真实 LLM API 接入**。

原因有四点：

1. 它对题目里“生成”二字的补强最直接。
2. 它对答辩时“这到底是不是 RAG”的说服力提升最大。
3. 它对第 2/3 章的方法与实现叙述最有实质增益。
4. 它是三项里唯一一个论文收益明显高于界面层收益的补丁方向。

但必须强调：
如果只补这一个，也应补成**最小 LLM 接入版**，而不是顺手把 Prompt 链和前端框架一起重构。

## 6. 最终建议

在当前冻结范围和收尾目标下，推荐的正式决策是：

1. **真实 LLM API：可做可不做，但若不做需在论文中改口径。**
2. **分步 Prompt：不必补做。**
3. **React + Tailwind：不必补做。**

如果后续确实决定动手，唯一值得优先考虑的是“最小 LLM API 接入”。
如果后续不动手，那么论文必须统一改口径为：

- 当前系统已完成 retrieval、evidence gating、answer assembly
- “生成”部分收口为证据驱动的规则化回答编排
- Prompt Engineering 和 React + Tailwind 不作为当前已完成实现宣称

## 7. 一句话版本

这轮不应该为了“看起来更像现代项目”去重写前端，也不应该为了“论文好看”硬塞多步 Prompt。真正值得认真权衡的只有真实 LLM API，而且也只值得做最小接入版。
