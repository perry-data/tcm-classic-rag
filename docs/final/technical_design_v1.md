## 1. 设计目标

当前技术设计的目标是在本科毕设范围内，完成一个高性能、高准确性且可演示的中医经典研读支持系统：

1. **混合检索闭环**：实现 FTS5 稀疏检索与 FAISS 稠密检索的深度融合。
2. **证据驱动生成**：确保所有回答内容均由真实证据槽位驱动，并利用大模型（LLM）进行文本润色。
3. **会话持久化**：支持多轮交互历史的存储与回溯。
4. **架构一致性**：确保实物代码、文档说明与开题报告的方法路线高度统一。

## 2. 系统总体架构

系统分为离线构建链路与在线运行链路。

### 2.1 离线构建链路 (Data Pipeline)

```text
原始回查底本 (ZIP)
  -> 数据清洗与结构化
  -> SQLite 数据库构建 (含 FTS5 索引)
  -> 文本嵌入 (Embedding)
  -> FAISS 向量索引构建
```

### 2.2 在线运行链路 (Runtime Pipeline)

```text
用户提问 (Query)
  -> 现代汉语语义解析
  -> 混合检索引擎 (Hybrid Engine)
     -> 稀疏召回 (FTS5/BM25)
     -> 稠密召回 (FAISS/BGE)
     -> 排名融合 (RRF)
     -> 交叉熵重排序 (BGE-Reranker)
  -> 证据门控 (Evidence Gating)
     -> 三槽位分类 (Primary/Secondary/Review)
  -> 回答编排器 (Answer Assembler)
     -> 规则化模版填充
     -> LLM 文本改写 (Qwen-plus)
  -> 结果返回 (Answer Payload)
     -> 渲染至聊天式前端
     -> 存入会话数据库 (Chat History DB)
```

## 3. 核心模块详细设计

### 3.1 混合检索引擎 (R1)
- **稀疏检索**：基于 SQLite FTS5，采用 `trigram` 分词，确保对中医专有名词（如方名、药名）的精确匹配。
- **稠密检索**：采用 `BAAI/bge-small-zh-v1.5` 模型进行文本向量化，弥补语义鸿沟。
- **重排序**：引入 `BAAI/bge-reranker-base` 对融合后的前 N 条结果进行精排，显著提升 Top-1 准确率。

### 3.2 证据门控逻辑 (R2)
严格执行证据分类协议：
- 只有置信度达标且属于 `main_passages` 的内容可入选 **Primary Evidence**。
- 注解类内容强制归入 **Secondary Evidence**。
- 若核心证据缺失，系统自动降级为 `weak` 模式或 `refuse`（拒答）模式。

### 3.3 LLM 接入方案 (R3)
- **模型**：阿里云通义千问 `qwen-plus`。
- **定位**：LLM 仅作为“文本美容师”，负责根据证据槽位内容生成流畅的 `answer_text`。
- **安全性**：通过 Prompt Engineering 严禁模型在证据之外自由发挥，确保“研读支持”而非“模型幻觉”。

### 3.4 存储层设计
- **主数据库**：`zjshl_v1.db`（只读，存储经典条文与索引）。
- **历史库**：`chat_history_v1.db`（读写，存储 Conversation 与 Message 对象）。

## 5. 系统规格与限制

- **语料范围**：当前仅覆盖《伤寒论》单书。
- **响应速度**：混合检索 + LLM 改写全过程平均耗时 < 3s。
- **环境要求**：Python 3.12+, 需配置 `TCM_RAG_LLM_API_KEY` 环境变量以启用 LLM。
