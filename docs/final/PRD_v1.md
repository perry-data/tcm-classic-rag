## 1. 项目背景

本课题定义为“基于检索增强生成的中医经典研读支持系统设计与实现”，其核心价值在于围绕中医经典研读过程，辅助用户通过现代汉语提问，获取可核对、有出处、有证据分层的解释结果。

这一目标对本科毕设场景尤其重要，原因有三：
1. **语义桥接**：弥补文言文经典与现代汉语提问之间的语义鸿沟。
2. **证据消除幻觉**：通过检索增强 (RAG) 确保回答均有据可查，避免大模型的知识幻觉。
3. **研读导向**：系统设计侧重于证据分层与溯源，而非简单的问答。

当前项目已完成从开题阶段到 V1.0 演示版的落地，形成了完整的研读支持链路：
`query -> hybrid retrieval -> evidence gating -> answer assembler (with LLM) -> POST /api/v1/answers -> 聊天式前端 (带历史记录)`

## 2. 产品目标

### 2.1 总目标
构建一个面向《伤寒论》单书场景的研读支持系统，使用户能够通过现代汉语提问，获得基于书内证据组织的回答、引用和出处线索。

### 2.2 V1.0 版本目标 (可答辩状态)
1. **主链路闭环**：稳定跑通基于 RAG 和证据分层的研读支持流程。
2. **真实 LLM 接入**：利用大语言模型（如通义千问 Qwen-plus）对基于证据生成的回答文本进行润色与改写。
3. **证据分层可视化**：清晰呈现主证据、辅助依据与参考资料，支持最小溯源。
4. **会话管理**：支持历史会话的持久化与查看，提升用户交互体验。

### 2.3 产品目标边界
系统定位于“研读支持”，明确排除临床诊疗、处方建议及双书扩展（如《金匮要略》）等非当前版本目标。

## 3. 目标用户
1. **中医药专业学生**：辅助条文定位、语义理解与课程学习。
2. **教师与研究者**：快速核对依据，演示系统化、可溯源的经典研读能力。
3. **一般研读爱好者**：辅助经典阅读，定位原文。

## 4. 典型使用场景
- **条文定位**：查询特定方剂或证候的原文。
- **语义理解**：针对经典句式的现代汉语解释。
- **总括研读**：对某一病证（如“太阳病”）的系统性分支整理。
- **比较研读**：对比不同方剂或证候的异同。
- **拒答场景**：对书外或无关问题明确拒答，保证学术严肃性。

## 5. 功能范围

### 5.1 正式功能模块 (V1.0)
1. **混合检索引擎**：结合 BM25 与向量检索，支持 RRF 融合与重排序。
2. **证据分层逻辑**：自动将候选证据分入 `primary / secondary / review` 三个槽位。
3. **回答编排器**：支持强回答、弱回答、拒答三种模式，并调用 LLM 进行文本优化。
4. **真实 LLM 接入**：已接通 Model Studio (Qwen-plus)，负责回答文本的生成与改写。
5. **聊天式前端**：支持 Markdown 渲染、证据分区展示、改问建议及**历史会话管理**。
6. **API 服务**：提供稳定的 RESTful 接口及流式 (Stream) 输出接口。

### 5.2 扩展功能 (已实现但不作为冻结主链核心)
- **历史记录存储**：基于 SQLite 的会话持久化。
- **流式回答响应**：提升前端交互的实时感。

## 6. 非目标范围 (当前版本)
1. 临床辨证论治与处方剂量建议。
2. 多书扩展（如《金匮要略》）。
3. 复杂知识图谱与本体推理。
4. 生产级高并发运维支撑。

## 7. 典型问题类型与支持度
| 问题类型 | 示例 | 支持强度 |
| --- | --- | --- |
| 条文出处 | `黄连汤方的条文是什么？` | 极高 (Strong) |
| 含义解释 | `烧针益阳而损阴是什么意思？` | 高 (LLM 改写支持) |
| 总括泛问 | `太阳病应该怎么办？` | 中 (分支整理) |
| 方剂比较 | `桂枝汤与麻黄汤的区别？` | 中 (Pairwise) |
| 无关拒答 | `量子纠缠是什么？` | 高 (Refuse) |

## 8. 成功标准 (答辩验收)
1. **闭环完整性**：从输入到输出、引用展示的全流程稳定。
2. **证据可溯源**：所有核心观点均能回指到《伤寒论》原文。
3. **口径一致性**：系统实物、文档说明与答辩演示内容完全匹配。
4. **性能基准**：检索响应时间在可接受范围内（< 2s），LLM 生成响应稳定。
