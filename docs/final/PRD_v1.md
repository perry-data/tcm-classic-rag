# 基于检索增强生成的中医经典研读支持系统 PRD v1

- 文档版本：v1
- 文档日期：2026-04-06
- 约束来源：`docs/proposal/221030147张前_开题报告.docx` + 当前仓库真实实现状态
- 文档定位：产品目标、用户价值、范围边界与成功标准的正式说明

## 1. 项目背景

开题报告将本课题定义为“基于检索增强生成的中医经典研读支持系统设计与实现”，核心价值不是让模型自由回答，而是围绕中医经典研读过程，帮助用户用现代汉语提问后获得可核对、有出处、有证据分层的解释结果。

这一目标对本科毕设场景尤其重要，原因有三点：

1. 《伤寒论》《金匮要略》等经典语言古奥，用户常常知道自己想问什么，但不知道该从哪一条、哪一卷、哪一类证候入手。
2. 通用大模型虽然能生成流畅回答，但若缺乏可回指证据，容易出现“像懂了、其实错了”的幻觉式解释。
3. 毕设系统要证明的不是“模型很聪明”，而是“系统能否把经典研读支持这件事做得更可检索、更可溯源、更可演示”。

当前项目已经从开题阶段的双书设想收口到《伤寒论》单书 MVP，形成了可运行主链路：

`query -> hybrid retrieval -> evidence gating -> answer assembler -> POST /api/v1/answers -> 同源前端单页渲染`

因此，本 PRD 的任务不是重新发散产品想象，而是把开题报告中的产品承诺，收束为当前可交付、可验证、可答辩的正式产品定义。

## 2. 产品目标

### 2.1 总目标

构建一个面向《伤寒论》单书场景的研读支持系统，使用户能够通过现代汉语提问，获得基于书内证据组织的回答、引用和出处线索，并在证据不足时看到弱回答或拒答，而不是无依据强答。

### 2.2 当前版本目标

当前版本的直接目标不是覆盖全部中医经典研读需求，而是验证以下命题成立：

1. 单书场景下，基于检索增强和证据分层的研读支持闭环可以稳定跑通。
2. 用户问题可以被归入不同问题类型，并得到不同强度的结果输出。
3. 系统能够把“回答”和“依据”分开呈现，支持最小溯源。
4. 在证据不足时，系统可以收缩为弱回答或拒答，从而守住非诊疗、证据优先的定位。

### 2.3 产品目标边界

当前产品目标明确是“研读支持”，不是：

- 临床诊断支持
- 处方推荐系统
- 中医知识图谱系统
- 多书统一问答平台
- 通用医学助手

## 3. 目标用户

当前目标用户按优先级分为三类：

1. 中医药相关专业学生
   需要围绕《伤寒论》条文、方名、证候、语句含义做课程学习、作业准备和答辩演示。
2. 教师与研究者
   需要快速核对条文依据、展示出处、演示“现代汉语提问 -> 书内证据回答”的系统能力。
3. 一般研读型用户
   对《伤寒论》有阅读兴趣，希望快速定位原文和相关解释，但不把系统当作医疗建议来源。

目标用户的共同特征是：

- 接受“经典研读”而非“临床定案”的系统定位。
- 需要证据、出处和引用，而不是只看一句结论。
- 可以接受系统在证据不足时不作确定性回答。

## 4. 典型使用场景

1. 条文定位场景
   用户问“黄连汤方的条文是什么？”，希望直接看到方文条目及出处。
2. 语义理解场景
   用户问“烧针益阳而损阴是什么意思？”，希望看到解释线索，并知道哪些只是辅助材料。
3. 总括研读场景
   用户问“太阳病应该怎么办？”，希望系统不要只返回一条，而是按几种典型分支整理。
4. 比较研读场景
   用户问“桂枝加附子汤方和桂枝加厚朴杏子汤方的区别是什么？”，希望看到配伍和条文语境差异。
5. 无依据问题场景
   用户问“书中有没有提到量子纠缠？”，系统应拒答并提示如何改问。

## 5. 核心用户问题

当前产品要解决的核心问题是：

1. 用户知道问题，但不知道对应哪一条经典原文。
2. 用户能提出现代汉语问题，但经典原文是文言文，直接查找成本高。
3. 仅靠大模型自由回答，缺少经典出处支撑，容易错引和过度概括。
4. 研读支持需要“答案 + 依据 + 引用 + 风险提示”同时成立，不能只输出流畅文本。
5. 当问题过宽或证据不足时，系统需要知道何时收缩，而不是假装全都能答。

## 6. 功能范围

### 6.1 当前正式功能范围

当前版本正式包含以下能力：

1. 单书范围内的现代汉语问题输入
2. 基于混合检索的候选证据召回
3. 主证据、补充依据、核对材料三槽位分层
4. `strong / weak_with_review_notice / refuse` 三模式输出
5. 条文/出处类问题处理
6. 含义解释类问题处理
7. 总括类/泛问类问题的最小分支整理
8. 两个方名之间的最小比较类问答
9. `POST /api/v1/answers` 最小接口
10. 同源前端单页展示回答、依据、引用和改问建议

### 6.2 当前已实现但需带条件理解的能力

1. “基于证据的生成式回答”
   当前落地为证据驱动的回答编排，而不是外接 LLM 的自由生成。
2. “现代汉语问法支持”
   当前对条文问、解释问、比较问、总括问已有最小覆盖，但语义解析仍偏启发式。
3. “效果验证”
   当前已完成功能 smoke、案例 smoke、前端联调 smoke 和数据验收；尚未形成大规模指标化问答评测集。

## 7. 非目标范围

以下内容当前明确不做：

1. 诊断、辨证论治、处方推荐、剂量建议
2. 多书扩展到《金匮要略》《黄帝内经》等
3. 知识图谱、本体系统、复杂关系推理
4. 多轮会话记忆、历史记录、用户系统
5. 生产级高并发、复杂部署与运维
6. 复杂后台管理系统
7. 非 pairwise comparison 的复杂多对象比较
8. 优劣判断类问题，如“哪个好”“谁更适合”

## 8. 当前版本边界

### 8.1 内容边界

- 当前正式语料只覆盖《伤寒论》。
- 当前数据库统计显示 `books = 1`，因此论文与答辩表达不能再按“双书已完成”叙述。

### 8.2 输出边界

- 输出是“研读支持答案”，不是“临床结论”。
- 回答可信度首先取决于是否能回指到证据，而不是表述是否流畅。

### 8.3 工程边界

- 当前前端是同源单页 MVP，不是 React + Tailwind 工程化前端。
- 当前回答层是规则化编排，不是接入大模型 API 的在线生成模块。

## 9. 典型问题类型分层

| 问题类型 | 典型示例 | 当前支持度 | 当前输出目标 |
| --- | --- | --- | --- |
| 条文/出处类 | `黄连汤方的条文是什么？` | 已处理较好 | 返回 `strong`，主依据清晰，引用稳定 |
| 含义解释类 | `烧针益阳而损阴是什么意思？` | 部分处理 | 优先弱回答，保留辅助材料和核对提示 |
| 泛问/总括类 | `太阳病应该怎么办？` | 最小可用 | 按 2-4 条分支整理，证据不足时降级 |
| 比较类 | `A 和 B 的区别是什么？` | 最小可用 | 当前只支持两个方名的最小比较 |
| 无证据问题 | `书中有没有提到量子纠缠？` | 已处理较好 | 明确拒答并给改问建议 |

当前处理得较好的，是“条文/出处类”和“无证据拒答类”。  
当前仍存在明显缺口的，是“复杂解释类”“超宽泛总括类”和“超出双方名范围的比较类”。

## 10. 系统输出与交互形式

当前系统输出不是一个自由文本块，而是一份稳定 answer payload，前端围绕以下内容渲染：

1. `answer_text`
2. `primary_evidence`
3. `secondary_evidence`
4. `review_materials`
5. `citations`
6. `review_notice`
7. `refuse_reason`
8. `suggested_followup_questions`

交互形式的核心要求有三条：

1. 用户必须能分清“系统说了什么”和“系统依据什么这么说”。
2. 主依据、补充依据、核对材料必须分区显示。
3. 拒答是合法业务结果，不是接口异常。

## 11. 成功标准

一个合格的毕设级研读支持系统，在当前版本中至少应满足：

1. 能在 Web 页面中稳定完成“输入问题 -> 返回回答 -> 查看依据 -> 查看引用”闭环。
2. 能明确展示其非诊疗定位。
3. 能对至少三类代表性问题给出不同强度的输出：强回答、弱回答、拒答。
4. 能在典型样例中保持证据分层不混乱，尤其不能把低层材料伪装成主证据。
5. 能对总括类和比较类问题展示“当前已有最小支持”，同时不过度夸大其完备性。
6. 文档、实现和答辩口径彼此一致，不出现“文档说有、系统其实没有”的硬伤。

## 12. 当前限制与风险

1. 当前仅覆盖《伤寒论》，与开题报告中的双书设想不一致。
2. 当前未接入真实 LLM 生成模块，因此“生成式回答”实现强度低于开题报告原设想。
3. 当前已正式落地 SQLite FTS5/BM25，但 `trigram` tokenizer 与 sparse score 映射仍按单书库调优。
4. 总括类和比较类能力已经可演示，但仍大量依赖启发式规则，不适合夸大为“完整专题整理能力”。
5. 当前验证主要是功能 smoke 与典型案例回归，尚未形成 200-250 条金标准测试集与指标化报告。
6. `annotation_links` 仍默认禁用，因此注解自动挂接能力尚未恢复。

## 14. 当前收口策略与后续推进原则

### 14.1 当前收口策略

1. 先守住单书、证据优先、非诊疗三条底线。
2. 先把“能稳定演示的最小闭环”做实，再谈更大范围的模型能力。
3. 遇到开题报告与当前实现不一致时，优先保留产品目标，收缩实现范围，而不是伪装成已完成。

### 14.2 后续推进原则

1. 若扩展到多书，必须先重写范围边界、评估口径和引用规则。
2. 若接入真实 LLM，必须继续保留当前证据门控和拒答机制，不能回到自由作答。
3. 若补做 FTS5/BM25、自建测试集、Prompt 工程，应被视为“补齐开题承诺”，而不是推翻当前正式系统。
4. 所有后续扩展都必须继续服务于“研读支持”，而不是滑向“诊疗系统”。
