# 《伤寒论》RAG 项目分层启用策略

## 1. 文档目的

本文件用于冻结《伤寒论》RAG 项目的数据分层启用规则。目标不是重新解析数据，也不是直接进入数据库或检索实现，而是明确：

- 哪些数据可以参与召回；
- 哪些数据可以作为主证据展示；
- 哪些数据只能作为辅助材料；
- 哪些数据只能后台参与候选筛选；
- 哪些数据在当前阶段仍然默认禁用。

本策略同时使用两份数据包的角色分工：

- `zjshl_dataset_v2_mvp_safe.zip`：运行时默认基线；
- `zjshl_dataset_v2.zip`：补充召回与风险回退层。

## 2. 为什么不能直接全量启用

不能直接把 `zjshl_dataset_v2.zip` 全量无条件启用，原因有三点：

1. `annotation_links` 已在验收中确认存在错挂，若直接启用，会把错误注解伪装成“已建立依据”的证据链。
2. `ambiguous_passages` 对应内容规模较大，且已有相当部分进入主条与切片层，若不分层，系统会把低置信度内容与高置信度正文混用。
3. `passages` 是全量总账，不是天然适合作为前台主证据的精炼证据层，其中混有正文、注解、附录、方剂组成、方法段等多种角色。

因此，直接全量启用会破坏“证据溯源优先”的项目定位。

## 3. 为什么也不应停留在 safe 版极保守模式

只停留在 safe 版的极保守模式，同样不是最佳状态，原因也很明确：

1. safe 版默认停用了 `annotation_links`，同时隔离了大量风险材料，这有利于安全，但会牺牲一部分召回覆盖面。
2. 某些用户问题更容易先命中注解表达、全文总账或低置信度片段；若这些材料完全不参与召回，系统会过早拒答。
3. 本项目是“研读支持系统”，允许弱回答与待核对提示，因此可以让风险材料参与候选召回，但不能让它们越级成为强证据。

因此，本项目不采用“全量无条件启用”，也不采用“safe 之外全部彻底禁用”，而采用“召回放宽、证据分级、展示受控”的分层启用策略。

## 4. 总体原则

本策略遵循以下原则：

1. 召回层可以宽于证据层。
2. 主证据层必须优先使用 safe 版中低风险、已筛过的数据。
3. 辅助材料可以参与解释，但必须与主证据分区展示。
4. 风险材料可以参与候选召回，但默认不能直接作为主证据。
5. 当只有风险材料命中时，系统必须进入弱回答、待核对提示或拒答模式。
6. 系统始终定位为《伤寒论》研读支持系统，不是诊疗系统。

## 5. 四层方案

本项目采用四层启用方案：

| 层级 | 名称 | 定位 | 是否参与召回 | 是否可作主证据 | 是否可前台展示 |
| --- | --- | --- | --- | --- | --- |
| Level A | 主证据层 | 默认强证据 | 是 | 是 | 是 |
| Level B | 辅助证据层 | 可解释、可补充，但不替代主证据 | 是 | 否 | 是，需分区或提示 |
| Level C | 候选召回层 | 后台扩召回与风险回退 | 是 | 否 | 默认否，仅风险模式可有限展示 |
| Level D | 禁用层 | 当前阶段不进入运行链路 | 否 | 否 | 否 |

## 6. 当前数据对象映射

### 6.1 Level A：可直接作为主证据展示

- `zjshl_dataset_v2_mvp_safe.zip / main_passages`
  条件：`retrieval_primary = true`
  用法：最终主依据、回答引用、正文级证据展示。

### 6.2 Level B：可作为辅助证据展示

- `zjshl_dataset_v2_mvp_safe.zip / main_passages`
  条件：`retrieval_primary = false`
  说明：这部分通常是 safe 版保留但降级的短主条。可作为辅助说明或补充引文，不作为默认主证据。

- `zjshl_dataset_v2.zip / annotations`
  条件：只按“注解原文片段”使用，不依赖 `annotation_links` 自动挂接。
  说明：可参与召回，也可在前台“参考材料/辅助说明”区展示，但必须加上“注解原文，未启用自动挂接”的提示，不能包装成已验证正文依据。

### 6.3 Level C：可参与召回，但默认不直接展示为主证据

- `zjshl_dataset_v2_mvp_safe.zip / chunks`
  角色：高质量召回骨架、命中片段预览。
  说明：虽然 safe `chunks` 质量较高，但它仍是检索单元而不是最终规范证据，因此归入 Level C 的“检索优先层”，命中后必须回指 `main_passages`。

- `zjshl_dataset_v2.zip / passages`
  角色：全量文本总账、全文兜底召回、后端回溯与上下文补全。
  说明：默认不直接作为主证据展示。只有在风险模式下，才允许以“待核对材料”身份有限展示。

- `zjshl_dataset_v2.zip / ambiguous_passages`
  角色：低置信度风险清单。
  说明：允许参与低权重召回或风险提示判断，不允许作为主证据。

- `zjshl_dataset_v2.zip` 中与 `ambiguous_passages` 关联的主条、切片、总账记录
  角色：风险候选材料。
  说明：可作为后台候选，用于判断“是否存在相关讨论”，但默认不进入主证据与普通辅助证据层。

### 6.4 Level D：默认禁用

- `zjshl_dataset_v2.zip / annotation_links`
  说明：当前阶段默认禁用，不参与召回、不参与证据展示、不参与回答拼接。

- 基于 `annotation_links` 自动生成的“正文-注解挂接证据链”
  说明：在 link 层通过重新验收前，整条链路都视为禁用。

- 原 v2 中被 safe 版排除的 `chunks` 作为独立索引对象
  说明：这些内容若需要参与召回，应由 `passages` / `annotations` / `ambiguous_passages` 层承接，不再以独立 `chunks` 身份启用。

## 7. 各阶段启用规则

### 7.1 检索阶段

检索阶段允许使用：

- Level A：safe `main_passages(retrieval_primary=true)`
- Level B：safe `main_passages(retrieval_primary=false)`、原始 `annotations`
- Level C：safe `chunks`、原始 `passages`、`ambiguous_passages` 关联内容

检索阶段不使用：

- Level D：`annotation_links` 及其派生链路

检索阶段的默认权重顺序应为：

1. safe `chunks`
2. safe `main_passages(retrieval_primary=true)`
3. safe `main_passages(retrieval_primary=false)`
4. raw `annotations`
5. raw `passages`
6. ambiguous 关联内容

### 7.2 证据筛选阶段

证据筛选阶段只允许：

- 从 Level A 中选主证据；
- 从 Level B 中选辅助证据；
- 把 Level C 仅保留为“候选参考”；
- Level D 完全不进入筛选结果。

必须满足以下规则：

1. 没有 Level A，不得进入“强证据回答”。
2. Level B 可以补充解释，但不能替代 Level A。
3. Level C 不能直接升格为主证据。
4. 若只有 Level B / Level C 命中，系统只能弱回答、待核对提示或拒答。

### 7.3 回答生成阶段

回答生成阶段建议将输入材料分成三个槽位：

- `primary_evidence`
  只放 Level A。

- `secondary_evidence`
  只放 Level B。

- `risk_materials`
  只放 Level C。

回答规则：

1. 只有 `primary_evidence` 时，可以输出标准回答。
2. `primary_evidence + secondary_evidence` 同时存在时，可以输出较完整回答，但要在展示层区分“主依据”和“辅助说明”。
3. 只有 `secondary_evidence` 时，只能输出弱回答，并提示“正文强证据不足”。
4. 只有 `risk_materials` 时，只能输出“待核对提示”或拒答，不得生成确定性解释。

## 8. 前台展示规则

### 8.1 可以展示给用户的内容

- Level A：可以展示为“主依据”。
- Level B：可以展示为“辅助说明”或“参考材料”。

### 8.2 只能后台参与候选召回的内容

- Level C 默认只参与召回、聚类、补充上下文和风险判断，不直接作为常规展示材料。

### 8.3 必须带风险提示的内容

以下内容一旦进入前台，必须显式标注：

- 注解原文片段
  标签建议：`注解参考（未启用自动挂接）`

- 低置信度材料或其关联片段
  标签建议：`低置信度 / 待核对`

- 只有辅助/风险材料支撑的回答
  标签建议：`强证据不足`

### 8.4 仍然暂不启用的内容

- `annotation_links`
- 所有基于 `annotation_links` 的自动证据链展示

## 9. 六类数据对象的明确使用结论

### 9.1 `passages` 到底怎么用

- 作用：全量总账、全文兜底召回、后端回溯、上下文补全。
- 层级：默认 Level C。
- 前台策略：默认不作主证据展示。

### 9.2 `main_passages` 到底怎么用

- 作用：主证据层核心对象。
- 层级：
  safe 且 `retrieval_primary=true` 为 Level A；
  safe 且 `retrieval_primary=false` 为 Level B；
  被 safe 排除的风险主条不直接进入 A/B。

### 9.3 `chunks` 到底怎么用

- 作用：第一阶段召回骨架、命中预览对象。
- 层级：safe `chunks` 为 Level C 中的高优先级检索层。
- 限制：`chunks` 本身不是最终规范证据，命中后应回指 `main_passages`。

### 9.4 `annotations` 是否可参与召回、是否可参与展示

- 可参与召回：是。
- 可参与展示：可以，但只作为 Level B 辅助材料展示。
- 限制：展示时必须明确标注为注解原文片段，不能冒充已验证正文依据。

### 9.5 `annotation_links` 是否仍默认禁用、未来灰度启用前提是什么

- 当前状态：仍默认禁用，属于 Level D。
- 未来灰度启用前提：
  1. 已确认错挂记录全部修正；
  2. 对疑似批量错挂记录完成抽样复核或重建；
  3. 新一轮验收中，严重错挂清零；
  4. 至少形成“可启用 link 的可信子集”后，才允许灰度上线。

### 9.6 `ambiguous_passages` 关联内容是否可参与召回、是否可当主证据

- 可参与召回：可以，但应低权重。
- 是否可当主证据：不可以。
- 前台策略：只有在风险模式下，才允许作为“待核对材料”有限展示。

### 9.7 当只有风险材料命中时，系统回答策略应该是什么

- 只能进入弱回答、待核对提示或拒答模式。
- 不得输出确定性结论。
- 页面层必须把“主依据不足”与“参考材料命中”明确区分。

## 10. 推荐的四种回答模式

| 模式 | 进入条件 | 允许使用的数据层 | 前台呈现 |
| --- | --- | --- | --- |
| Strong | 至少 1 条 Level A 主证据 | A，可附带 B | 主依据 + 可选辅助说明 |
| Strong+Aux | A 与 B 同时存在 | A + B | 主依据区 + 参考说明区 |
| Weak+Review | 无 A，但有 B 或 C 的相关候选 | B 或 C | 弱回答 + 待核对提示 |
| Refuse | 没有可形成稳定解释的 A/B，且 C 也不成簇 | 无 | 拒答或请求用户缩小问题范围 |

## 11. 结论

本项目后续实现应采用“双数据源、分层启用”的策略：

- 运行时基线使用 `zjshl_dataset_v2_mvp_safe.zip`；
- 原始 `zjshl_dataset_v2.zip` 只承担补充召回与风险回退角色；
- `chunks` 负责召回；
- `main_passages` 负责最终主证据；
- `annotations` 只作为带标签的辅助材料；
- `annotation_links` 继续默认禁用；
- `ambiguous_passages` 关联内容只进入低权重召回和风险模式，不进入主证据层。

这套策略能够兼顾三点：

1. 不把未验明的数据直接推上前台；
2. 不把 safe 版之外的内容彻底浪费；
3. 保持“证据分级、可弱答、可待核对”的研读支持系统定位。
