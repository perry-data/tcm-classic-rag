## 1. 决策前提

当前项目已经具备三个关键条件：

1. 主链闭环已经稳定，不需要再通过“继续堆功能”证明系统存在。
2. 150 条 evaluator v1 与 qwen-plus live validation 已经把当前系统的稳定边界立住。
3. 接下来的核心问题不是“还能做多少东西”，而是“下一轮只做一个方向时，做什么最值”。

因此，本决策遵守四条规则：

1. 下一轮只允许有一条主线，不做多线并行。
2. 优先选择对现有稳定边界破坏最小的方向。
3. 优先选择能为后续所有优化提供判断依据的方向。
4. 优先选择能同时提升论文收口质量与答辩说服力的方向。

## 2. P0 / P1 / P2 排序

### P0：最应该先做

#### 评估体系优化

判断理由：

1. 当前 evaluator v1 已证明系统“稳定可用”，但还不能精细证明“哪一层优化最值得做”。
2. retrieval、generation、performance 三条线下一步都会遇到同一个问题：没有更细的 before/after 指标，就无法证明优化收益。
3. 当前 qwen-plus live 已成功，但 answer_text 质量没有正式评分规则；当前 retrieval 已稳定，但没有 Hit@K / Recall@K / rerank 对比；当前性能也没有 p50 / p95 基线。
4. 评估体系优化不直接破坏正式主链，风险最低，却能为后续所有主线建立统一尺子。
5. 对论文第 4 章收益最大，对答辩时“你如何证明优化有效”这一问题最有帮助。

结论：

> 当前唯一应该先做的主线是：**评估体系优化**。

### P1：第二梯队

#### 检索质量优化

排序理由：

1. 检索质量决定证据质量上限，是后续生成质量的基础。
2. 相比前端和性能，它更接近论文题目中的核心方法贡献。
3. 但它不适合先于评估体系优化启动，否则容易陷入“参数调了，但收益不清楚”的状态。

#### Prompt / 生成质量优化

排序理由：

1. qwen-plus live 验证已经成功，说明这条线具备现实可做性。
2. 生成优化能显著提升最终 answer_text 的自然度、结构性和答辩展示观感。
3. 但它依赖 answer_text 质量评价与回归基线，不能在当前直接抢跑。

P1 内推荐顺序：

1. 检索质量优化
2. Prompt / 生成质量优化

原因：

- 检索质量优化改善的是证据基础。
- 生成质量优化改善的是证据之上的表达层。
- 证据层应先于表达层。

### P2：可以做，但当前不应作为主线

#### 性能优化

排序理由：

1. 当前尚无明确瓶颈与正式 benchmark。
2. 现在最需要的是先测量，而不是先调优。
3. 性能相关工作更适合作为评估体系优化中的一个子模块先落地。

#### 前端展示优化

排序理由：

1. 当前前端已满足毕设演示的基本闭环。
2. 展示优化能加分，但不改变检索增强系统的研究核心说服力。
3. 若过早推进，容易把精力从论文主线移到 UI 打磨。

P2 内推荐顺序：

1. 性能优化
2. 前端展示优化

## 3. 第一轮唯一优化目标

### 3.1 明确结论

第一轮唯一优化目标明确选定为：

> **评估优化**

不是检索优化，不是生成质量优化，不是前端优化，也不是性能优化。

### 3.2 为什么它应该先于其他项

#### 为什么先于检索优化

因为当前检索已经进入“可用且稳定”阶段，但还缺少 retrieval 级诊断指标。  
如果现在直接做检索优化，最容易发生的情况是：

1. 某些样例变好；
2. 另一些样例变差；
3. 总体是否真的更好却没有统一证据。

也就是说，检索优化现在最缺的不是“能不能改”，而是“有没有更精细的尺子去证明改得值”。

#### 为什么先于生成质量优化

因为 qwen-plus live validation 已证明 LLM 路径真实可用，但当前还没有正式 answer_text 质量评价体系。  
如果现在直接做 prompt 调优，结果很容易停留在：

1. 主观觉得更顺；
2. 但缺少题型化、可复核的评分结果；
3. 难以在论文中严谨说明“优化前后差异”。

因此，生成质量优化的前提是先把评价尺子建出来。

#### 为什么先于前端优化

因为当前前端虽不华丽，但已经稳定消费冻结 payload 并可完整演示。  
前端优化对观感有帮助，但不会回答检索是否更准、答案是否更好、系统是否更可靠这些核心问题。

#### 为什么先于性能优化

因为当前性能方向最大的问题不是“慢点已经定位清楚”，而是“还没有正式 benchmark 证据”。  
当前更合理的动作是先把 latency baseline 纳入评估体系，而不是先单独开一条性能调优主线。

### 3.3 第一轮目标的具体定义

第一轮“评估优化”不是泛泛做评估，而是收束为四个可执行子任务：

1. 把 retrieval 指标从最终 citation 正确扩展到 `Hit@K / Recall@K / rerank 前后对比 / 题型分组统计`。
2. 为 qwen-plus 路径建立 answer_text 质量评价规约，至少覆盖清晰度、结构性、证据一致性、弱答边界保持。
3. 把 latency mini-benchmark 纳入同一评估工件，形成 baseline 与 LLM-enabled 的可比较结果。
4. 建 failure taxonomy，把“为什么错”系统化，而不是只看最终 pass/fail。

这四项仍属于同一条主线，因为它们服务的是同一目标：

> 让后续优化从“凭感觉推进”，升级为“可度量、可比较、可复盘地推进”。

## 4. 推荐执行顺序

在不多线并行的前提下，推荐顺序如下：

1. 评估优化
2. 检索优化
3. 生成质量优化
4. 性能优化
5. 前端优化

这个顺序背后的逻辑是：

1. 先把尺子做细。
2. 再优化证据基础。
3. 再优化答案表达。
4. 再根据 benchmark 决定是否值得调性能。
5. 最后再做展示层润色。

## 5. 当前不值得做的优化项

以下事项当前不值得优先推进：

### 5.1 React + Tailwind 前端重写

不值得现在做，原因是：

1. 收益主要集中在工程风格和视觉现代化，而不是研究核心。
2. 会引入新的联调成本与演示风险。
3. 不利于论文收口，适合写入展望而非当前主线。

### 5.2 多步 Prompt 编排

不值得现在做，原因是：

1. 当前单步 prompt 已经稳定 live-call。
2. answer_text 评价体系还未补齐。
3. 多步 prompt 会先带来复杂度、延迟和调试成本。

### 5.3 多书扩展

不值得现在做，原因是：

1. 会破坏单书《伤寒论》冻结边界。
2. 会连带重开数据、引用、检索、评估、论文口径五个层面。
3. 更适合写进系统展望。

### 5.4 在评估尺子未升级前继续扩 goldset 数量

不值得现在做，原因是：

1. 当前最缺的是“评估更细”，不是“题量更大”。
2. 直接扩到 200-250 更像堆量，不像解决当前最关键的度量问题。

### 5.5 生产级性能/并发/部署优化

不值得现在做，原因是：

1. 当前系统定位是本地演示型 MVP。
2. 这类工作会明显偏离论文核心贡献。
3. 收益与当前阶段成本不成比例。

### 5.6 修改 payload contract 或 evidence 分层规则

不值得现在做，原因是：

1. 当前稳定性最宝贵的部分正是冻结的合同与分层语义。
2. 贸然改动会破坏 qwen-plus live validation、150 evaluator 与前端渲染的一致性。

## 6. 最终决策

本轮决策的最终结论只有一条：

> 在当前系统已经稳定、150 条评估已通过、qwen-plus live 已验证、主链已经闭环的前提下，下一轮唯一应该先推进的优化目标是：**评估优化**。

它先于检索优化、先于生成质量优化、先于前端优化、也先于独立性能优化。

只有先把评估体系从“基础可用”升级到“能够精细判断下一步收益”，后续的任何优化主线才不会变成盲调。
