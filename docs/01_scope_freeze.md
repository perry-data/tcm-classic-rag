# 毕业设计 MVP 范围冻结文档

## 1. 项目目标

本项目拟实现一个面向《伤寒论》文本研读场景的最小可用系统，用于支持用户围绕条文内容进行检索、提问、回答与依据查看。系统采用单体 Web 应用形态，优先完成从“用户提问”到“返回带依据的回答”的 MVP 闭环。

本项目的核心目标不是构建通用中医知识平台，也不是构建医疗诊断系统，而是验证“基于检索增强生成与证据溯源的经典文本研读支持”这一思路在单书场景中的可行性。

本文件用于冻结本阶段范围。除非后续单独修订，否则后续开发、任务拆解与论文需求分析均应以本文件为准，不应自行扩展需求。

## 2. 项目要解决的问题

围绕《伤寒论》进行文本研读时，用户通常面临以下问题：

- 条文数量较多，人工查找相关原文效率较低。
- 用户提出自然语言问题后，难以快速定位可作为依据的相关条文。
- 直接使用大模型回答时，容易出现脱离原文、依据不清或无法溯源的问题。
- 在证据不足的情况下，系统若仍强行作答，会削弱研读支持的可信度。

因此，本项目要解决的不是“自动给出医学结论”，而是“如何在《伤寒论》单书范围内，为研读问题提供可检索、可回答、可查看依据、证据不足时可拒答”的基础能力。

## 3. 目标用户

本项目的目标用户限定为与《伤寒论》研读有关的使用者，主要包括：

- 需要围绕条文内容进行学习、查阅和对照的中医药相关专业学生。
- 在课程学习、读书讨论或毕业设计演示中，需要快速查看条文依据的教师或研究者。
- 对《伤寒论》文本内容有研读需求、但不追求临床决策支持的普通使用者。

目标用户默认具备基本中文阅读能力，并接受系统输出应以原文依据为中心，而非以结论权威性为中心。

## 4. MVP 必做功能

本阶段必须完成以下功能，缺一不可：

### 4.1 单书语料范围固定

- 系统语料仅包含《伤寒论》。
- 系统中的检索、问答与依据展示均只能基于该书内容展开。
- 不接入其他中医典籍、教材、网络资料或外部知识库。

### 4.2 面向条文内容的提问输入

- 用户可以通过 Web 页面输入与《伤寒论》条文相关的问题。
- 问题形式可以是自然语言问题，也可以是对某一概念、症状描述或条文含义的查询。
- MVP 不要求支持复杂对话编排，单轮问答可用即可。

### 4.3 相关条文检索

- 系统能够针对用户问题返回相关条文或文本片段。
- 检索结果应作为后续回答生成的依据输入，而不是仅返回最终答案。
- 用户至少应能看到与问题相关的条文内容，而不是只看到模型生成结果。

### 4.4 基于依据的回答生成

- 系统在回答问题时，应尽量基于检索到的《伤寒论》条文内容组织答案。
- 回答应保持“研读支持”定位，重点是对条文内容进行归纳、解释或对应说明。
- 回答不得表述为医疗诊断、处方建议、剂量建议或治疗方案推荐。

### 4.5 原文依据展示与溯源

- 系统回答应尽量附带可见的原文证据或条文依据。
- 依据展示至少应包含条文文本本身；如实现条件允许，可附带条文编号或片段位置。
- 用户应能明确区分“系统生成的回答”和“支持该回答的原文依据”。

### 4.6 证据不足时的弱回答或拒答

- 当检索结果不足以支持较明确回答时，系统应允许输出“证据不足”的提示。
- 系统可以给出弱回答，但必须显式说明依据有限，不能伪装为确定性结论。
- 在无法形成可靠依据时，拒答优先于强行作答。

### 4.7 基础 Web 闭环

- 系统以单体 Web 应用方式提供基本可用界面。
- 用户至少能够完成“输入问题、查看回答、查看依据”这一最小交互流程。
- 本阶段不要求完整产品化体验，但必须具备可演示、可验证的使用闭环。

## 5. MVP 明确不做的内容

以下内容明确排除在本阶段范围之外，后续开发不得默认纳入：

- 多书扩展，包括《金匮要略》《黄帝内经》等其他典籍接入。
- 多模态能力，包括图片、音频、手写稿识别或扫描版解析。
- 知识图谱、本体构建、术语网络或复杂关系推理系统。
- 模型微调、领域继续预训练或专用大模型训练。
- 多智能体协作框架、复杂工作流编排或自动研究代理。
- 医疗诊断、辨证论治、处方推荐、剂量建议、问诊流程等诊疗能力。
- 面向生产环境的大规模并发、高可用部署、分布式架构与复杂运维体系。
- 复杂用户体系，包括细粒度权限、多角色后台、审计系统等。
- 花哨前端、重交互可视化、个性化推荐或与研读闭环无关的展示功能。
- 在本轮提前细化数据库表结构、接口代码、检索算法实现、大模型接入代码和测试脚本。

## 6. 系统边界说明

### 6.1 内容边界

- 系统只处理《伤寒论》文本及其可被引用的条文片段。
- 系统不对书外知识负责，也不承诺覆盖中医经典的完整知识范围。

### 6.2 功能边界

- 系统负责“检索相关文本、组织回答、展示依据、在证据不足时收缩回答”。
- 系统不负责“替代教师讲解、替代人工研读、给出临床决策”。

### 6.3 输出边界

- 输出应被视为研读辅助信息，而不是医疗建议。
- 输出可信度以是否能回溯到原文依据为主要判断标准。
- 当原文依据不足时，系统可以不回答完整结论。

### 6.4 工程边界

- 本阶段以单体 Web 应用为准，不拆分为微服务或多进程复杂系统。
- 只追求 MVP 可用，不追求高性能、高并发和深度优化。
- 若新需求不能直接服务于最小闭环，应默认延后处理。

### 6.5 需求控制边界

- 开发过程中不得因为“技术上可以实现”而自动加入额外功能。
- 开题报告、讨论记录或临时想法中的未冻结设想，不能直接视为本阶段必做项。
- 若确需新增需求，应先判断其是否属于 MVP 闭环刚需；若不是，则归入后续阶段。

## 7. 成功标准

本项目在 MVP 层面达到以下标准，即可认为“范围内任务完成”：

- 用户能够在 Web 界面输入与《伤寒论》相关的问题。
- 系统能够返回与问题相关的条文或文本片段。
- 系统能够基于检索结果生成与条文内容相关的回答。
- 回答中能够看到明确的原文依据，用户可以区分答案与证据。
- 当依据不足时，系统能够做出弱回答或拒答，而不是无依据地强答。
- 整个流程能够围绕“经典研读支持”成立，并在演示中清楚体现其非诊疗定位。

本阶段的成功不以功能数量多少为标准，而以是否形成“单书、可检索、可问答、可溯源、可收缩”的最小闭环为标准。

## 8. 风险与约束

### 8.1 数据与文本约束

- 《伤寒论》文本版本、条文切分方式和清洗质量会直接影响检索与依据展示效果。
- 若文本切分不稳定，可能导致回答能生成但依据定位不清，削弱系统可信度。

### 8.2 模型与回答约束

- 大模型仍可能出现过度概括、脱离原文或语言过强的问题。
- 因此，本项目必须坚持“依据优先、证据不足可拒答”的约束，不以回答流畅度代替可靠性。

### 8.3 项目范围约束

- 本项目是本科毕业设计，时间、人力和工程资源有限。
- 若过早扩展到多书、多模块或复杂架构，极易导致实现分散，削弱 MVP 完成度。

### 8.4 论文表达约束

- 论文中对系统能力的表述应与本文件一致，避免将“研读支持”写成“智能诊疗”。
- 如需描述创新点，应聚焦“检索增强生成 + 证据溯源 + 范围可控的单书场景验证”，而非夸大系统智能程度。

## 9. 后续开发阶段建议

后续开发可按以下阶段推进：

1. 语料整理与条文切分阶段
2. 检索与证据组织阶段
3. 问答闭环打通阶段
4. Web 演示与交互完善阶段
5. 论文整理与结果验证阶段
