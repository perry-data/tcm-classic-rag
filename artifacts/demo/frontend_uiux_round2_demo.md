# Frontend UI/UX Round 2 Demo

## 演示目标

本轮演示重点不是新接口，而是“同一套接口下，页面是否更清楚、更稳、更像正式系统”。

## 改前 / 改后对比

### 改前

- 结果模式主要靠 badge 区分
- `strong / weak / refuse` 的含义需要读多块区域才能理解
- 错误态容易和拒答态混淆
- review 材料和主依据并列出现，主次关系还不够明显
- 完成后 progress 仍比较显眼

### 改后

- 回答区顶部新增“结果状态头”
- strong 会显示“可直接参考的回答”
- weak 会显示“需核对的回答”
- refuse 会显示“当前不支持这样回答”
- error 会显示“本次请求未完成”，并明确说明“这不是拒答”
- 主依据更突出，review 材料默认折叠
- 完成后 progress 会降权显示，不再抢占主要视线

## 四类场景演示

### 1. Strong

query：

- `黄连汤方的条文是什么？`

改后界面重点：

- 顶部结果状态头显示“可直接参考的回答”
- 回答区提示先看正文，再核对主依据
- 主依据区最突出
- 回答引用区会说明“这些引用与当前回答和证据区块直接对应”

### 2. Weak With Review Notice

query：

- `烧针益阳而损阴是什么意思？`

改后界面重点：

- 顶部结果状态头显示“需核对的回答”
- 回答区明确提示“不能视为确定结论”
- `review_notice` 仍单独高亮
- 补充依据可见，但核对材料默认收起，避免压过回答正文

### 3. Refuse

query：

- `书中有没有提到量子纠缠？`

改后界面重点：

- 顶部结果状态头显示“当前不支持这样回答”
- 页面会明确说明“这属于正常业务拒答，不是系统报错”
- 拒答原因与改问建议仍单独展示
- 不会误导用户把拒答当成请求失败

### 4. Error

测试方式：

- 模拟流式 transport failure
- 随后同步回退也失败

模拟结果：

- 顶部结果状态头切换为“本次请求未完成”
- mode badge 切换为“请求异常”
- 错误区出现：
  - 标题：`流式与标准请求都未完成`
  - 错误说明
  - “这不是拒答”的提示
  - “重新提交上一个问题”按钮
- 进度区停止在中断态，不再残留“正在生成”视觉状态

## 本轮验证摘要

- 页面结构检查：
  - `mode-summary` 已真实输出
  - `error-section` 已真实输出
  - `review-toggle` 已真实输出
  - `Cmd/Ctrl + Enter` 提示已真实输出
- 后端联调检查：
  - `strong` 仍返回 `strong`
  - `weak_with_review_notice` 仍返回 `weak_with_review_notice`
  - `refuse` 仍返回 `refuse`
- 前端错误态模拟检查：
  - 错误区出现
  - retry 按钮可用
  - progress 不残留 active 生成态
